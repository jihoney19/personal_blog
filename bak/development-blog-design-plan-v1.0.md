# 개인 개발 블로그 디자인 기획 문서 v1.0

## 0. 문서 정보

| 항목 | 내용 |
|---|---|
| 문서 상태 | 사용자 승인 내용을 반영한 디자인 기획 확정본 |
| 작성일 | 2026-07-15 |
| 대상 | Astro 기반 개인 개발 블로그 로컬 MVP |
| 기반 문서 | `개발블로그_기획문서_v1.0.md`, `DESIGN-opencode.ai.md` |
| 디자인 기준 | OpenCode.ai의 터미널 미학을 한국어 개발 블로그에 맞게 완화하여 적용 |
| 구현 전제 | 본 문서 승인 후 화면 시안과 `AGENTS.md`를 확정하고 구현 시작 |

이 문서는 색상, 글꼴, 레이아웃, 반응형 규칙, 컴포넌트 상태와 화면 구성을 정의한다. 서비스 목표, 콘텐츠 데이터 구조와 기능 범위는 서비스·기능 기획 문서를 따르며, 구현 절차와 QA 명령은 별도 `AGENTS.md`에서 정의한다.

### 0.1 관련 문서 정합성 메모

항목별 승인 과정에서 서비스·기능 기획 초안보다 구체화되거나 변경된 다음 결정은 차기 서비스·기능 기획 문서에도 동기화해야 한다.

- 카테고리는 `학습 메모·문제 해결·기술 아티클·프로젝트 회고` 같은 글 형식이 아니라 `AI Agent·프론트엔드·백엔드·Astro` 같은 기술 주제로 정의한다.
- 현재 MVP에서는 이메일과 연락하기 기능을 제외하고 GitHub 링크만 외부 연락 수단으로 제공한다.

두 문서가 동기화되기 전에는 위 항목에 한해 사용자와 나중에 합의한 본 디자인 문서의 결정을 적용한다.

## 1. 디자인 콘셉트

### 1.1 핵심 방향

- OpenCode.ai의 터미널·manpage 미학을 핵심 레퍼런스로 사용한다.
- 크림색 캔버스, 잉크색 텍스트, 모노스페이스 영문 UI, ASCII 브래킷과 얇은 구분선을 정체성 요소로 사용한다.
- 그림자, 그라데이션, 유리 효과와 장식용 이미지는 사용하지 않는다.
- 채용 담당자와 비개발자 협업자도 읽는 한국어 블로그이므로, 본문 가독성은 레퍼런스보다 우선한다.
- 다크 터미널 히어로는 홈에서 한 번만 사용하고 일반 콘텐츠 영역에는 반복하지 않는다.
- 여백과 정렬로 위계를 만들며, 카드가 떠 보이는 입체 효과는 사용하지 않는다.

### 1.2 형태 원칙

| 대상 | 모서리 | 표현 |
|---|---:|---|
| 글 목록, 섹션, 프로젝트 카드, 본문 컨테이너 | `0px` | 배경 또는 1px hairline으로 구분 |
| 버튼, 배지, 입력창, 코드 복사 버튼 | `4px` | 작고 절제된 인터랙티브 형태 |
| 프로필 이미지 | 원형 | 사용자 프로필에만 제한적으로 사용 |

`6~8px` 반경의 일반 카드 스타일은 사용하지 않는다.

### 1.3 ASCII 브래킷 사용 원칙

`[+]`, `[-]`, `[x]`는 화면 구조를 알려주는 브랜드 표식으로만 사용한다.

**사용 위치**

- 주요 섹션 라벨
- 글 유형 프리픽스
- 목차 열기·닫기
- 상태 또는 빈 상태의 짧은 제목

**사용하지 않는 위치**

- 일반 본문 문단
- Markdown 글머리표와 번호 목록
- 모든 메뉴, 링크와 버튼
- 반복되는 메타정보

브래킷은 장식이 아니라 정보 구조를 빠르게 인식시키는 제한된 표식이어야 한다.

## 2. 타이포그래피

### 2.1 글꼴 구성

| 용도 | 기본 글꼴 | 대체 글꼴 |
|---|---|---|
| 영문 UI, ASCII 요소, 코드 | Berkeley Mono | JetBrains Mono, IBM Plex Mono, `ui-monospace`, Consolas, monospace |
| 한국어 제목과 본문 | Pretendard | `-apple-system`, BlinkMacSystemFont, "Segoe UI", "Noto Sans KR", sans-serif |

- Berkeley Mono는 정식 사용 권한이 있는 웹폰트 파일을 제공받았을 때 사용한다.
- Berkeley Mono를 제공할 수 없는 환경에서는 JetBrains Mono를 첫 번째 대체 글꼴로 사용한다.
- 한국어를 모노스페이스 글꼴로 강제하지 않는다.
- 코드블록과 터미널 히어로는 언어와 관계없이 모노스페이스를 유지한다.

### 2.2 타입 스케일

| 역할 | 데스크톱 | 모바일 | 굵기 | 줄간격 |
|---|---:|---:|---:|---:|
| 홈 히어로 | `38px` | `28px` | 700 | 1.5 |
| 글 상세 제목 | `32px` | `26px` | 700 | 1.4 |
| 본문 H2 | `24px` | `22px` | 700 | 1.5 |
| 본문 H3 | `20px` | `19px` | 700 | 1.5 |
| 섹션 라벨 | `16px` | `16px` | 700 | 1.5 |
| 한국어 본문 | `17px` | `16px` | 400 | 1.8 |
| UI·버튼 | `16px` | `16px` | 500 | 1.5 |
| 메타정보·캡션 | `14px` | `14px` | 400 | 1.7 |
| 코드 | `15px` | `14px` | 400 | 1.7 |

- 한국어 본문의 최대 폭은 `720px`로 제한한다.
- 긴 영문 URL과 코드가 레이아웃을 넓히지 않도록 별도 overflow 규칙을 적용한다.
- 강조를 위해 기울임꼴에만 의존하지 않고 굵기 또는 구조를 함께 사용한다.

## 3. 컬러와 테마

### 3.1 기본 컬러 토큰

| 토큰 | 라이트 | 다크 | 용도 |
|---|---|---|---|
| canvas | `#fdfcfc` | `#201d1d` | 페이지 배경 |
| ink | `#201d1d` | `#fdfcfc` | 제목과 고강조 텍스트 |
| body | `#424245` | `#d5d2d2` | 본문 텍스트 |
| mute | `#646262` | `#aaa6a6` | 메타정보와 보조 텍스트 |
| surface-soft | `#f8f7f7` | `#2a2626` | 약한 보조 배경 |
| surface-card | `#f1eeee` | `#302c2c` | 코드블록과 입력 요소 |
| hairline | `rgba(15,0,0,0.12)` | `rgba(255,255,255,0.14)` | 기본 구분선 |
| hairline-strong | `#646262` | `#aaa6a6` | 선택 상태와 강한 구분선 |
| accent | `#005fcc` | `#6ea8ff` | 링크, 포커스, 선택 상태 |
| success | `#18794e` | `#32d583` | 성공·완료 |
| warning | `#8a4b00` | `#f5a524` | 주의·진행 중 |
| danger | `#b42318` | `#ff6b6b` | 오류·실패 |

- 일반 화면은 canvas와 ink 중심의 모노크롬 인상을 유지한다.
- 상태 색상은 장식에 사용하지 않는다.
- 상태는 색상만으로 전달하지 않고 `완료`, `주의`, `오류` 같은 문구나 아이콘을 함께 제공한다.
- 실제 구현 시 WCAG 2.2 AA 대비 검사를 통과한 값만 확정 토큰으로 사용한다.

### 3.2 테마 동작

1. 첫 방문에는 운영체제의 라이트·다크 설정을 따른다.
2. 사용자가 테마 버튼을 누르면 라이트와 다크가 전환된다.
3. 사용자 선택은 브라우저에 저장하며 새로고침과 재방문 시 유지한다.
4. 저장된 선택이 있으면 운영체제 설정보다 사용자 선택을 우선한다.
5. 초기 렌더링 전에 테마를 적용해 화면이 순간적으로 반전되는 현상을 방지한다.
6. 테마 버튼의 접근성 이름은 실행할 동작을 설명한다. 예: `다크 모드로 전환`.

### 3.3 코드 문법 강조 팔레트

| 역할 | 라이트 | 다크 |
|---|---|---|
| 코드 배경 | `#f1eeee` | `#302c2c` |
| 기본 코드 | `#201d1d` | `#fdfcfc` |
| 주석 | `#646262` | `#aaa6a6` |
| 키워드·함수 | `#005fcc` | `#6ea8ff` |
| 문자열 | `#18794e` | `#6ee7a0` |
| 숫자·상수 | `#8a4b00` | `#ffc66d` |
| 오류 강조 | `#b42318` | `#ff6b6b` |

- 문법 강조는 제한된 색상만 사용하며 무지개색 표현을 피한다.
- 긴 코드는 자동 줄바꿈하지 않고 가로 스크롤한다.
- 색상 없이 읽어도 토큰과 코드 구조를 파악할 수 있어야 한다.

## 4. 레이아웃과 반응형

### 4.1 너비와 여백

| 항목 | 값 |
|---|---:|
| 전체 콘텐츠 최대 폭 | `1100px` |
| 일반 페이지 본문 최대 폭 | `960px` |
| 글 상세 본문 최대 폭 | `720px` |
| 글 상세 목차 | 약 `200px` |
| 목차와 본문 사이 간격 | `40px` |
| 데스크톱 좌우 여백 | `32px` |
| 태블릿 좌우 여백 | `24px` |
| 모바일 좌우 여백 | `16px` |

### 4.2 섹션 리듬

- 데스크톱: 주요 섹션 사이 `96px`
- 태블릿: 주요 섹션 사이 `64px`
- 모바일: 주요 섹션 사이 `48px`
- 컴포넌트 내부 간격은 8px 기반의 `4 / 8 / 12 / 16 / 24 / 32px` 체계를 사용한다.

### 4.3 브레이크포인트

| 구간 | 폭 | 주요 변화 |
|---|---|---|
| desktop-large | `1280px+` | 최대 폭 레이아웃, 왼쪽 sticky 목차 |
| desktop | `1024~1279px` | 데스크톱 구조 유지 |
| tablet | `851~1023px` | 좌우 여백과 섹션 간격 축소 |
| tablet-compact | `769~850px` | 목차를 본문 상단 접이식으로 전환 |
| tablet-narrow | `641~768px` | 상단 메뉴를 모바일 메뉴로 전환 |
| mobile | `640px 이하` | 완전한 단일 컬럼, 히어로와 타입 크기 축소 |

최소 검수 폭은 `360px`로 한다.

## 5. 공통 탐색 영역

### 5.1 상단 메뉴

- 높이 `56px`, canvas 배경, 하단 1px hairline을 사용한다.
- 화면 상단에 고정하며 본문을 가리지 않도록 레이아웃 공간을 확보한다.
- 왼쪽에는 블로그 이름 `DEV.LOG`을 표시한다.
- 데스크톱 오른쪽에는 `홈 · 카테고리 · 태그 · 프로젝트 · 소개`를 표시한다.
- 가장 오른쪽에는 라이트·다크 테마 버튼을 둔다.
- 현재 페이지는 글자 굵기와 하단선으로 구분한다. 브래킷을 추가하지 않는다.
- 본문 바로가기를 위한 `콘텐츠로 건너뛰기` 링크를 키보드 포커스 시 노출한다.

**모바일**

- 블로그 이름, 테마 버튼과 메뉴 버튼만 상단에 표시한다.
- 메뉴 버튼을 누르면 전체 메뉴가 펼쳐진다.
- 메뉴가 열리면 버튼의 펼침 상태를 보조기기에 전달하고, `Esc`로 닫을 수 있게 한다.
- 메뉴가 닫힐 때 포커스를 메뉴 버튼으로 되돌린다.

### 5.2 푸터

- 상단 1px hairline과 `32px` 안쪽 여백을 사용한다.
- `DEV.LOG`과 한 줄 소개를 표시한다.
- `GitHub · 프로젝트 · 소개` 링크를 제공한다.
- 현재 MVP에는 이메일 또는 연락하기 기능을 구현하지 않는다.
- 실제 주소가 없는 외부 링크는 표시하지 않는다.
- 하단에 저작권 연도와 실명을 표시한다.
- 모바일에서는 링크를 한 줄씩 세로 배치한다.
- 개인정보처리방침과 이용약관은 현재 로컬 MVP 범위에 포함하지 않는다.

## 6. 콘텐츠 분류의 시각적 구분

콘텐츠 유형, 카테고리와 태그의 역할을 겹치지 않게 구분한다.

| 구분 | 의미 | 예시 | 기본 표현 |
|---|---|---|---|
| 콘텐츠 유형 | 글의 형식 | 학습 메모, 문제 해결, 기술 아티클, 프로젝트 회고 | 제목 앞 브래킷 프리픽스 |
| 카테고리 | 글의 대표 기술 주제 | AI Agent, 프론트엔드, 백엔드, Astro | 메타정보 또는 필터 탭 |
| 태그 | 세부 기술과 키워드 | Codex, TypeScript, 접근성 | 본문 하단 텍스트 링크 |

- 글마다 콘텐츠 유형과 카테고리는 각각 하나를 사용한다.
- 태그는 여러 개를 허용한다.
- 홈의 필터는 카테고리 기준으로 동작한다.
- 유형은 정렬 순서에 영향을 주지 않는다.

## 7. 페이지별 디자인

### 7.1 홈

**구성 순서**

1. 다크 터미널 히어로
2. 블로그 소개문구
3. 카테고리 필터
4. 최신 글 목록
5. 게시글 빈 상태

**다크 터미널 히어로**

- `$ whoami` 형식으로 실명 또는 정체성을 표현한다.
- 핵심 문구는 `AI Agent를 활용해 아이디어를 실제 서비스로 완성하는 풀스택 개발자` 방향을 사용한다.
- 배경은 surface-dark, 텍스트는 크림색으로 구성한다.
- ASCII 요소와 모노스페이스 영문을 사용하되, 한국어 설명은 Pretendard로 읽기 쉽게 표시한다.
- 글 데이터에 의존하지 않아 게시글이 없어도 완성된 첫 화면을 유지한다.

**게시글이 없는 초기 상태**

- 히어로와 소개문구는 정상 노출한다.
- 게시글이 하나도 없으면 카테고리 필터를 숨긴다.
- 글 목록 위치에 `[+] 새로운 기록을 준비하고 있습니다`를 표시한다.
- 안내 문구는 다음을 사용한다.

> 아직 작성된 글이 없습니다.  
> 새로운 학습 기록을 준비하고 있습니다.

- 샘플 글, 가짜 기록과 비활성 카테고리 버튼은 표시하지 않는다.
- 첫 게시글이 추가되면 카테고리 필터와 최신 글 목록을 자동 노출한다.

### 7.2 글 목록 카드

```text
[학습 메모] AI Agent로 블로그 기획하기
2026-07-15 · AI Agent · 5분 읽기
요약 텍스트가 최대 두 줄까지 표시됩니다.
```

- 유형 프리픽스와 제목을 첫 영역에 표시한다.
- 제목은 최대 2줄, 요약은 최대 2줄로 제한한다.
- 날짜·카테고리·읽기 시간은 한 줄을 기본으로 하되 모바일에서는 줄바꿈을 허용한다.
- 목록은 콘텐츠 유형과 관계없이 게시일 최신순으로 정렬한다.
- 카드 배경과 그림자를 추가하지 않고 hairline과 여백으로 구분한다.
- 제목 링크의 클릭 영역을 카드 전체로 확장하되 중첩 링크를 만들지 않는다.
- hover와 키보드 포커스에서 hairline을 강하게 표시한다.
- 시각적으로 생략한 제목과 요약도 접근성 이름과 원문 데이터에서는 보존한다.

### 7.3 글 상세

```text
┌──────────────┬──────────────────────────────────┐
│ [+] 목차      │ 유형 · 카테고리                    │
│ 시작하며       │ 제목                              │
│ 설계 원칙      │ 작성일 · 수정일 · 읽기 시간          │
│ 마무리         │ 본문                              │
│              │ 태그                              │
│              │ 이전 글 / 다음 글                   │
│              │ 관련 글                            │
└──────────────┴──────────────────────────────────┘
```

**목차**

- 데스크톱에서는 왼쪽 약 `200px` sticky 사이드바로 항상 표시한다.
- 본문 제목이 0개 또는 1개여도 목차 컨테이너를 숨기지 않는다.
- 생성할 목차 항목이 없으면 `목차 항목이 없습니다.`를 표시한다.
- 스크롤 위치에 따라 현재 섹션을 굵기와 왼쪽선으로 강조한다.
- `850px` 이하에서는 본문 제목 아래, 본문 시작 전의 `[+] 목차` 접이식으로 전환한다.
- 태블릿과 모바일에서는 기본적으로 닫힌 상태로 시작한다.

게시글 자체가 없으면 글 상세 페이지는 생성되지 않으므로 홈의 게시글 빈 상태만 표시한다.

**헤더와 메타정보**

- 1행: 콘텐츠 유형과 카테고리
- 2행: 글 제목
- 3행: 작성일·수정일·예상 읽기 시간
- 태그는 본문 하단에 배치한다.
- 수정일이 없으면 해당 항목과 구분점을 함께 숨긴다.

**코드블록**

- 상단 왼쪽에 언어 이름을 표시한다.
- 상단 오른쪽에 `복사` 버튼을 둔다.
- 복사 성공 시 버튼 문구를 2초 동안 `복사됨`으로 변경한다.
- 복사 실패 시 `복사하지 못했습니다`를 표시하고 다시 시도할 수 있게 한다.
- 언어 정보가 없으면 언어 라벨 영역만 숨기고 복사 버튼은 유지한다.
- 데스크톱 `15px`, 모바일 `14px`, 줄간격 `1.7`을 사용한다.
- 긴 코드는 줄바꿈하지 않고 가로 스크롤한다.
- 복사 결과는 화면과 보조기기에 함께 알린다.

**이전 글·다음 글**

- 게시일을 기준으로 연결한다.
- 존재하는 글만 카드로 표시하며 빈 카드와 비활성 버튼은 만들지 않는다.
- 한쪽 글만 존재하면 해당 카드만 표시한다.

**관련 글**

- 같은 태그의 글을 우선하고 부족하면 같은 카테고리의 글로 보충한다.
- 현재 글은 제외하고 최대 3개 표시한다.
- 관련 글이 없으면 섹션 자체를 숨긴다.

### 7.4 카테고리와 태그

**카테고리 인덱스**

- 카테고리 이름순으로 표시한다.
- 각 카테고리의 게시글 개수를 함께 표시한다.

**태그 인덱스**

- 게시글이 많은 태그부터 표시한다.
- 동률이면 이름순으로 정렬한다.
- 각 태그의 게시글 개수를 함께 표시한다.

**상세 목록**

- 해당 카테고리 또는 태그의 게시글을 최신순으로 표시한다.
- 초기 MVP에서는 별도 정렬·필터 UI를 제공하지 않는다.
- 항목이 전혀 없는 인덱스 페이지에는 빈 상태 안내를 표시한다.
- 글이 없는 개별 카테고리·태그 상세 페이지는 생성하지 않는다.

### 7.5 프로젝트 목록

- 프로젝트명, 한 줄 설명, 상태, 담당 역할과 핵심 기술을 표시한다.
- 카드형이라는 이유로 그림자나 큰 반경을 추가하지 않는다.
- 상태는 색상과 텍스트를 함께 사용한다.
- 카드 전체를 내부 프로젝트 상세 페이지의 기본 링크로 사용한다.
- 첫 프로젝트로 실제 `개인 개발 블로그`를 표시한다.

### 7.6 프로젝트 상세

상단에 `cat project.json` 느낌의 스펙시트를 제공한다.

| 순서 | 필드 | 표시 규칙 |
|---:|---|---|
| 1 | 상태 | 필수, 텍스트와 상태색 함께 사용 |
| 2 | 개발 기간 | 시작일과 종료일 또는 진행 중 표시 |
| 3 | 담당 역할 | 사용자의 실제 역할 표시 |
| 4 | 주요 기술 | 핵심 기술만 요약 표시 |
| 5 | GitHub | 주소가 있을 때만 표시 |
| 6 | 데모 사이트 | 주소가 있을 때만 표시 |

- 데스크톱은 2열 정보표, 모바일은 1열로 표시한다.
- 영문 필드 라벨은 모노스페이스로 정렬한다.
- 저장소와 데모 주소가 없으면 해당 행과 버튼을 완전히 숨긴다.
- 스펙시트 아래에는 `배경 → 목표 → 역할 → 주요 기능 → 문제 해결 → QA 결과 → 회고` 순서의 서술형 콘텐츠를 배치한다.

**프로젝트 이미지**

- 대표 이미지는 선택 항목으로 둔다.
- 실제 화면 캡처는 본문 폭에 맞춰 크게 표시한다.
- 이미지 아래에 화면 설명과 캡션을 제공한다.
- 여러 이미지는 맥락을 따라 세로로 배치한다.
- 클릭 확대와 슬라이드 갤러리는 초기 MVP에서 제외한다.
- 의미 있는 이미지에는 대체 텍스트를 필수로 제공한다.
- 이미지가 없어도 완성된 정보 구조를 유지하며 가짜 플레이스홀더를 만들지 않는다.

### 7.7 소개

- 상단에 프로필 이미지, 실명과 대표 소개 문구를 표시한다.
- 데스크톱 상단은 이미지와 소개의 2열, 모바일은 이미지 위·소개 아래의 1열로 구성한다.
- 본문은 `관심 분야 → 기술 활용 방식 → AI Agent 협업 방식` 순서로 설명한다.
- 기술 스택은 게이지나 숙련도 막대로 나열하지 않고 프로젝트와 글로 증명한다.
- 하단에 대표 프로젝트와 대표 글을 배치한다.
- 현재 MVP의 외부 연락 수단은 GitHub 링크만 제공한다.
- 이메일과 연락하기 기능은 구현하지 않는다.
- 실제 정보가 없는 항목에는 임시 경력, 수치, 링크나 가짜 문구를 만들지 않는다.

### 7.8 404와 오류 상태

- 404 페이지는 `[x] 페이지를 찾을 수 없습니다`와 홈으로 이동하는 버튼을 제공한다.
- 필터 결과가 없으면 오류가 아니라 빈 상태로 안내한다.
- 코드 복사 실패 등 복구 가능한 오류는 해당 컴포넌트 가까이에 원인 없는 짧은 안내와 재시도 수단을 제공한다.
- 이미지 로드에 실패해도 대체 텍스트와 문서 흐름이 유지되어야 한다.
- 상태 메시지는 색상만으로 구분하지 않는다.

- 표와 코드블록은 모바일 레이아웃 전체가 넓어지지 않도록 자체 스크롤 영역을 갖는다.

### 9.1 레퍼런스 컴포넌트 매핑

| 레퍼런스 컴포넌트 | 블로그 적용처 |

| `hero-tui-mockup` | 홈의 `$ whoami` 히어로 |
| `primary-nav` | 상단 고정 메뉴 |
| `button-tab` / `button-tab-active` | 홈 카테고리 필터 |
| `badge-news` | 글 유형 라벨의 구조 참고 |
| `list-row` | 글 목록과 관련 글 |
| `faq-row` | 태블릿·모바일 접이식 목차 |
| `install-snippet` | 코드블록의 기본 표면 |
| `footer-section` | 개인 블로그용 간결한 푸터 |
| `hairline` | 섹션과 행 구분선 |

### 9.2 신규 컴포넌트

- 글 목록 카드
- 데스크톱 sticky 목차
- 태블릿·모바일 접이식 목차
- 코드 언어 라벨과 복사 버튼
- 빈 상태
- 이전 글·다음 글
- 관련 글
- 테마 전환 버튼
- 프로젝트 스펙시트
- 프로젝트 상태 배지
- 모바일 메뉴
- 404 상태

### 9.3 인터랙션 상태

- hover는 색상 또는 1px 선만 변경한다.
- active는 배경 또는 텍스트를 한 단계 진하게 표시한다.
- `:focus-visible`에는 2px accent 아웃라인과 충분한 간격을 제공한다.
- 카드가 떠오르거나 확대되는 효과는 사용하지 않는다.
- 기본 전환 시간은 약 `150ms`로 제한한다.
- 펼침 메뉴와 목차는 짧은 전환만 사용한다.
- 운영체제의 `prefers-reduced-motion` 설정이 활성화되면 불필요한 애니메이션을 제거한다.
- 버튼과 메뉴의 목표 영역은 최소 `44×44px`을 권장한다.

### 9.4 아이콘 원칙

- 구조 표식은 승인된 ASCII 브래킷을 사용한다.
- 테마, 메뉴, 복사, 외부 링크처럼 기능 이해에 필요한 경우에만 단순한 선형 아이콘을 사용한다.
- 장식용 아이콘과 이모지는 사용하지 않는다.
- 아이콘 단독 버튼에는 반드시 화면에 보이지 않는 접근성 이름을 제공한다.
- 동일 기능의 아이콘은 전 페이지에서 같은 모양과 크기를 사용한다.

## 10. 접근성

목표 수준은 WCAG 2.2 AA로 한다.

- 일반 텍스트는 최소 4.5:1, 큰 텍스트와 UI 경계는 최소 3:1 대비를 확보한다.
- 모든 기능을 키보드만으로 사용할 수 있어야 한다.
- 포커스 순서는 시각적 읽기 순서와 일치해야 한다.
- `:focus-visible`에서만 명확한 2px accent 아웃라인을 표시한다.
- 상단에 `콘텐츠로 건너뛰기` 링크를 제공한다.
- 페이지마다 하나의 H1을 사용하고 제목 수준을 건너뛰지 않는다.
- 의미 있는 이미지에는 대체 텍스트를 제공하고 장식 이미지는 보조기기에서 제외한다.
- 복사 결과와 메뉴 펼침 상태는 보조기기에 전달한다.
- 현재 위치, 선택 상태와 오류를 색상에만 의존해 표현하지 않는다.
- 텍스트를 200% 확대해도 콘텐츠와 기능이 손실되지 않아야 한다.
- 동작 줄이기 설정을 존중한다.

## 11. 조건부 표시 규칙

- 게시글이 없으면 홈의 카테고리 필터와 글 목록을 숨기고 빈 상태를 표시한다.
- 게시글 자체가 없으면 글 상세 페이지를 생성하지 않는다.
- 글 상세 페이지가 존재하면 목차 컨테이너는 제목 개수와 관계없이 항상 표시한다.
- `updatedAt`이 없으면 수정일과 인접 구분점을 숨긴다.
- 태그가 비어 있으면 태그 영역을 숨긴다.
- 관련 글이 없으면 관련 글 섹션을 숨긴다.
- 이전 글 또는 다음 글이 없으면 해당 카드만 숨긴다.
- `repoUrl`과 `demoUrl`이 없으면 관련 행과 버튼을 숨긴다.
- GitHub 주소가 없으면 소개와 푸터의 GitHub 링크를 숨긴다.
- 실제 프로필 이미지가 없으면 가짜 인물 이미지를 만들지 않는다.
- 초안 콘텐츠는 공개 목록과 정적 페이지 생성 대상에서 제외한다.

## 12. 화면 시안 제작 범위

구현 전에 다음 화면을 와이어프레임과 시각 목업으로 확인한다.

1. 홈 — 게시글이 없는 초기 상태
2. 홈 — 게시글과 카테고리 필터가 있는 상태
3. 글 상세 — 데스크톱 라이트 모드
4. 글 상세 — 모바일 다크 모드
5. 카테고리와 태그 인덱스
6. 프로젝트 목록
7. 프로젝트 상세
8. 소개
9. 모바일 메뉴
10. 404 페이지

초기 실제 사이트에는 샘플 게시글을 넣지 않는다. 목록과 상세 상태의 디자인 검증이 필요하면 배포 콘텐츠와 분리된 테스트 fixture만 사용한다.

## 13. 디자인 승인 기준

다음 조건을 모두 만족하면 디자인 단계를 승인 가능한 상태로 본다.

- OpenCode 레퍼런스의 터미널 정체성과 한국어 본문 가독성이 함께 유지된다.
- 브래킷이 정해진 구조 표식에만 사용된다.
- 라이트·다크 모드에서 텍스트와 상태의 명암 대비가 기준을 통과한다.
- 게시글이 없는 홈이 오류나 미완성 화면처럼 보이지 않는다.
- 글 상세의 목차가 제목 개수와 관계없이 정의된 방식으로 표시된다.
- 긴 제목, 본문, 코드, 표, 이미지와 URL이 레이아웃을 깨뜨리지 않는다.
- 데스크톱, 태블릿과 모바일에서 탐색·테마·목차·코드 복사를 사용할 수 있다.
- 키보드 탐색, 포커스, 대체 텍스트와 상태 안내가 설계에 포함된다.
- 존재하지 않는 링크, 연락처, 콘텐츠와 이미지를 임의로 만들지 않는다.
- 화면 간 컴포넌트, 간격, 글꼴, 색상과 상태 표현이 일관된다.

## 14. 승인된 의사결정 요약

| 주제 | 결정 |
|---|---|

| 목차 | 글 상세에서 항상 표시, 작은 화면에서는 상단 접이식 |
| 상단 메뉴 | 56px 고정 메뉴, 모바일 메뉴 제공 |
| 푸터 | DEV.LOG, 프로젝트·소개·GitHub, 연락하기 제외 |
| 테마 | 첫 방문은 운영체제 설정, 이후 사용자 선택 저장 |
| 상태 색상 | 텍스트와 함께 사용, 다크 모드 전용 값 제공 |
| 코드 강조 | 제한된 의미 색상, 긴 코드는 가로 스크롤 |
| 코드블록 | 언어 라벨, 복사 버튼과 성공·실패 안내 |
| 게시글 빈 상태 | 필터를 숨기고 실제 안내만 표시 |
| 카테고리·태그 | MVP에서 추가 정렬 UI 없이 고정 기준 적용 |
| 글 목록 | 제목 2줄, 요약 2줄, 최신순 |
| 이전·다음·관련 글 | 존재하는 항목만 조건부 표시 |
| 프로젝트 스펙시트 | 상태·기간·역할·기술·조건부 외부 링크 |
| 프로젝트 이미지 | 선택 항목, 세로 흐름, 캡션·대체 텍스트 제공 |
| 소개 | 서술형 소개와 GitHub만 제공, 연락하기 제외 |
| 본문 | 한국어 장문 중심 스타일, 표·코드 모바일 overflow 처리 |
| 반응형 | 1100px 프레임, 720px 본문, 96/64/48px 섹션 리듬 |
