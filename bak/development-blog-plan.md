1. 프로젝트 개요

| 항목 | 내용 |
|---|---|
| 프로젝트명 | 개인 개발 블로그 |
| 사이트 성격 | 개인 학습 기록 및 개발자 포트폴리오 |
| 핵심 메시지 | AI Agent를 활용해 아이디어를 실제 서비스로 완성하는 풀스택 개발자 |
| 주요 독자 | 본인, 동료 개발자, 채용 담당자 및 협업 관계자 |
| 기본 언어 | 한국어 |
| 사용자 공개 범위 | 실명 및 프로필 공개 |
| 디자인 기획 | 색상, 글꼴, 레이아웃, 컴포넌트와 화면 시안은 별도 문서로 작성 |
| 기술 스택 | Astro 기반 정적 사이트 |
| 현재 개발 범위 | 로컬 개발 및 검증까지 진행하며 외부에는 배포하지 않음 |

### 1.1 핵심 목표

1. 학습 과정과 문제 해결 경험을 부담 없이 기록한다.
2. 기술적 사고 과정과 프로젝트 경험을 포트폴리오로 보여준다.
3. 기획, 디자인, 개발, QA, 문서화에 AI Agent를 활용하는 프로젝트 운영 역량을 검증한다.
4. 게시글과 프로젝트를 Markdown 기반으로 쉽게 추가하고 관리할 수 있게 한다.

### 1.2 프로젝트 성공 기준

- 로컬 환경에서 모든 필수 페이지와 기능이 정상적으로 작동한다.
- 게시글이 없어도 완성도 있는 빈 상태 화면을 제공한다.
- 첫 번째 프로젝트로 이 개발 블로그의 상세 페이지를 제공한다.
- 사용자가 코드를 직접 검토하지 않아도 화면과 QA 결과로 승인할 수 있다.
- 새 Markdown 파일을 추가하면 별도 페이지 작업 없이 목록과 상세 페이지에 반영된다.
- 라이트 모드와 다크 모드, 모바일과 데스크톱 화면을 모두 지원한다.

## 2. 역할과 AI Agent 협업 방식

### 2.1 사용자 역할

사용자는 프로젝트 오너이자 최종 승인자 역할을 맡는다.

- 프로젝트 목표와 큰 방향 승인
- 단계별 결과물 확인 및 승인
- 실명, 프로필 이미지, 소개 등 본인 정보 제공
- 게시글과 프로젝트 콘텐츠의 최종 사실 확인
- 외부 배포, 비용 발생, 개인정보 처리 등 중요한 변경 승인

### 2.2 AI Agent 역할

AI Agent는 다음 업무를 주도적으로 수행한다.

- 요구사항 구체화 및 기획 문서 관리
- 정보 구조와 화면 흐름 설계
- UI 디자인 및 구현
- Astro 프로젝트 구조와 기능 개발
- 테스트, 오류 수정 및 회귀 검증
- 기술 문서와 운영 문서 작성
- 학습 주제 제안 및 우선순위 관리
- 학습 메모, 기술 글, 프로젝트 글 초안 작성
- 코드 예제와 기술적 사실 검증

### 2.3 운영 원칙과 문서 경계

- 승인된 단계 안의 세부 업무는 AI Agent가 수행하고 사용자는 큰 단계의 결과를 승인한다.
- 중요한 범위 변경, 외부 공개, 비용 발생, 보안 및 개인정보 문제는 별도 승인을 받는다.
- AI가 생성한 콘텐츠는 사용자의 실제 경험과 일치하는지 승인 후 사용한다.
- 존재하지 않는 경험, 성과, 수치, 링크는 임의로 만들지 않는다.
- 코딩 규칙, 파일 구조, 작업 절차, 테스트 방법, QA 체크리스트와 결과 보고 형식은 본 문서가 아닌 별도 `AGENTS.md`에서 정의한다.

### 2.4 단계별 승인 지점

| 단계 | 주요 결과물 | 승인 기준 |
|---|---|---|
| 1. 기획 | 기획 문서, 사이트 구조, 기능 범위, 완료 조건 | 목표와 범위가 명확하고 누락된 핵심 요구사항이 없음 |
| 2. 디자인 | 별도의 디자인 기획 문서와 주요 화면 시안 | 디자인 문서에 정의된 방향과 화면 구성이 적절함 |
| 3. 로컬 MVP | 로컬에서 작동하는 사이트와 핵심 기능 | 콘텐츠 관리, 탐색, 테마 전환이 요구사항대로 작동함 |
| 4. 최종본 | 전체 QA 결과와 수정 완료본 | 기능, 반응형, 접근성, 성능 검증을 통과함 |

단계 중간에는 범위 확대, 비용 발생, 보안 및 개인정보 문제처럼 사용자 판단이 필요한 경우에만 추가 승인을 요청한다.

## 3. 사이트 구조

```text
홈
├── 블로그 소개 문구
├── 카테고리 필터
├── 모든 글 최신순 목록
└── 게시글이 없을 때의 빈 상태 안내

글 상세
├── 제목, 요약, 작성일, 수정일
├── 콘텐츠 유형, 카테고리, 태그
├── Markdown 본문
├── 목차
├── 코드 문법 강조 및 복사 버튼
├── 이전 글·다음 글
└── 관련 글

카테고리
└── 카테고리별 글 목록과 글 개수

태그
└── 태그별 글 목록과 글 개수

프로젝트
├── 프로젝트 목록
└── 프로젝트 상세
    ├── 개요와 목표
    ├── 개발 기간 및 상태
    ├── 사용자와 AI Agent의 역할
    ├── 기술 스택과 선택 이유
    ├── 주요 기능
    ├── 문제 해결 과정
    ├── 결과와 회고
    └── 저장소, 배포 주소, 관련 글

소개
├── 실명과 프로필 이미지
├── 한 줄 소개
├── 관심 분야와 기술 스택
├── AI Agent 활용 방식
├── 대표 프로젝트와 글
└── GitHub 및 연락 수단
```

### 3.1 상단 메뉴

`홈`, `카테고리`, `태그`, `프로젝트`, `소개`를 기본 메뉴로 제공한다. 라이트·다크 모드 전환 기능도 탐색 영역에서 접근할 수 있어야 한다. 정확한 위치와 표현은 디자인 기획에서 결정한다.

## 4. 페이지별 상세 기획

### 4.1 홈

- 모든 게시글을 구분 없이 최신순으로 표시한다.
- 각 목록 항목에 제목, 작성일, 콘텐츠 유형, 카테고리, 요약, 예상 읽기 시간을 표시한다.
- 콘텐츠 유형은 라벨로 구분하되 노출 순서에는 영향을 주지 않는다.
- 상단에는 블로그의 목적을 설명하는 짧은 소개 문구와 카테고리 필터를 제공한다.
- 첫 로컬 버전에는 게시글을 넣지 않는다.
- 게시글이 없을 때는 다음과 같은 안내를 표시한다.

> 아직 작성된 글이 없습니다.  
> 새로운 학습 기록을 준비하고 있습니다.

### 4.2 글 상세

- 제목, 설명, 작성일, 수정일, 콘텐츠 유형, 카테고리, 태그를 표시한다.
- 긴 글을 위해 현재 위치를 파악하고 제목으로 이동할 수 있는 목차를 제공한다.
- 코드 블록에는 문법 강조, 언어 표시, 복사 버튼을 제공한다.
- 긴 코드와 표가 모바일 화면을 깨뜨리지 않도록 가로 스크롤을 지원한다.
- 같은 태그나 카테고리를 기준으로 관련 글을 제공한다.
- 이전 글과 다음 글은 게시일을 기준으로 연결한다.

### 4.3 카테고리와 태그

- 카테고리는 글의 대표 주제를 나타내며 글마다 하나를 사용한다.
- 태그는 기술과 세부 주제를 나타내며 여러 개를 사용할 수 있다.
- 각 카테고리와 태그에 해당하는 글 개수를 표시한다.
- 글이 없는 카테고리나 태그 페이지는 생성하지 않는다.

초기 카테고리 후보는 다음과 같다.

- 학습 메모
- 문제 해결
- 기술 아티클
- 프로젝트 회고
- AI Agent

### 4.4 프로젝트 목록

- 프로젝트명, 한 줄 설명, 상태, 담당 역할, 핵심 기술을 제공한다.
- 각 항목은 사이트 내부의 프로젝트 상세 페이지로 연결한다.
- 첫 번째 프로젝트로 `개인 개발 블로그`를 등록한다.

### 4.5 프로젝트 상세

모든 프로젝트는 동일한 구조의 상세 페이지를 사용하고, 내용은 AI Agent가 초안을 작성한 뒤 사용자가 승인한다.

- 프로젝트 배경과 해결하려는 문제
- 목표와 주요 사용자
- 개발 기간과 진행 상태
- 사용자의 역할과 기여 범위
- AI Agent가 담당한 업무
- 사용 기술과 선택 이유
- 핵심 기능과 화면
- 구현 중 발생한 문제와 해결 과정
- 테스트 및 QA 결과
- 결과, 아쉬운 점, 다음 개선 사항
- GitHub, 배포 사이트, 관련 블로그 글

`개인 개발 블로그` 프로젝트는 로컬 개발 중에는 `진행 중`으로 표시하며, 외부 링크가 없으면 해당 항목을 숨긴다.

### 4.6 소개

- 실명과 프로필 이미지를 공개한다.
- 대표 소개 문구는 다음 방향을 사용한다.

> AI Agent를 활용해 아이디어를 실제 서비스로 완성하는 풀스택 개발자

- 프론트엔드, 백엔드, 풀스택, AI·Agent 관심 분야를 하나의 흐름으로 설명한다.
- 기술 이름만 나열하지 않고 프로젝트와 관련 글로 역량을 증명한다.
- 이메일 주소는 화면에 그대로 노출하기보다 연락 버튼 형태를 우선 사용한다.
- 실제 이름, 사진, 소개, GitHub 주소, 연락처는 디자인 단계에서 제공받는다.

## 5. 콘텐츠 운영 기획

### 5.1 콘텐츠 유형

| 유형 | 목적 | 작성 수준 |
|---|---|---|
| 학습 메모 | 개념, 실험, 명령어, 참고 자료의 짧은 기록 | 짧거나 정리 중인 내용도 공개 가능 |
| 문제 해결 | 실제 오류와 해결 과정을 재사용 가능한 지식으로 정리 | 문제, 원인, 시도, 해결, 배운 점 포함 |
| 기술 아티클 | 하나의 기술 주제를 검증된 설명과 예제로 전달 | 코드와 기술적 사실 검증 필수 |
| 프로젝트 회고 | 프로젝트 목표, 역할, 결정, 성과와 개선점 기록 | 실제 경험과 기여 범위 중심 |

### 5.2 콘텐츠 원칙

- 학습 메모를 포함한 모든 글은 홈에서 동일하게 최신순으로 표시한다.
- 짧은 메모를 억지로 긴 글로 늘리지 않는다.
- 확인되지 않은 내용은 사실처럼 단정하지 않는다.
- 참고 자료를 사용한 경우 출처 링크를 남긴다.
- 코드 예제는 가능한 범위에서 실행하거나 테스트한 뒤 게시한다.
- AI가 작성한 초안은 사용자의 실제 학습과 경험을 반영하도록 최종 검토한다.

### 5.3 AI 기반 콘텐츠 작성 흐름

1. AI가 프로젝트 진행 상황과 포트폴리오 구성에 맞는 학습 주제를 제안한다.
2. 현재 수준, 실용성, 포트폴리오 가치를 기준으로 우선순위를 정한다.
3. 학습 자료와 실습 과제를 구성한다.
4. 실습 및 프로젝트 결과를 바탕으로 글 초안을 작성한다.
5. 코드, 링크, 기술적 사실을 검증한다.
6. 제목, 설명, 카테고리, 태그, 읽기 시간을 생성한다.
7. 사용자가 경험과 내용의 정확성을 승인한다.
8. 승인된 글만 게시 상태로 전환한다.

### 5.4 초기 콘텐츠 상태

- 첫 로컬 버전에는 게시글을 추가하지 않는다.
- 샘플 글이나 가짜 학습 기록을 만들지 않는다.
- 프로젝트에는 실제 개발 중인 `개인 개발 블로그`만 등록한다.

## 6. 콘텐츠 데이터 구조

Astro Content Collections를 사용해 글과 프로젝트를 Markdown 또는 MDX 파일로 관리한다. 데이터 필드는 개발 단계에서 스키마로 검증한다.

### 6.1 게시글 필드

| 필드 | 필수 | 설명 |
|---|---|---|
| `title` | 예 | 글 제목 |
| `description` | 예 | 목록과 검색 메타데이터에 사용할 요약 |
| `publishedAt` | 예 | 최초 게시일 |
| `updatedAt` | 아니요 | 주요 내용 수정일 |
| `type` | 예 | 학습 메모, 문제 해결, 기술 아티클, 프로젝트 회고 |
| `category` | 예 | 대표 카테고리 한 개 |
| `tags` | 예 | 세부 주제 태그 목록이며 빈 배열 허용 |
| `draft` | 예 | 초안 여부 |
| `status` | 아니요 | 학습 중, 검증 필요, 정리 완료 등의 표시 |

URL 주소는 파일명 또는 별도 슬러그를 기반으로 영문 소문자와 하이픈을 사용한다. 본문과 화면 표시는 한국어로 제공한다.

### 6.2 프로젝트 필드

| 필드 | 필수 | 설명 |
|---|---|---|
| `title` | 예 | 프로젝트명 |
| `summary` | 예 | 한 줄 설명 |
| `status` | 예 | 기획 중, 진행 중, 완료, 유지보수 중 |
| `startedAt` | 예 | 시작일 |
| `endedAt` | 아니요 | 종료일 |
| `role` | 예 | 사용자의 역할 |
| `stack` | 예 | 주요 기술 목록 |
| `repoUrl` | 아니요 | 소스 저장소 주소 |
| `demoUrl` | 아니요 | 배포 주소 |
| `relatedPosts` | 아니요 | 관련 글 목록 |

링크 값이 없으면 빈 버튼이나 준비 중 링크를 표시하지 않고 해당 UI 자체를 숨긴다.

### 6.3 URL 및 슬러그 규칙

- 슬러그는 영문 소문자와 하이픈(`-`)만 사용하며, 파일명 자체를 슬러그로 사용하는 것을 기본으로 한다.
- 필요한 경우 콘텐츠 파일의 frontmatter에 `slug` 필드를 별도로 지정해 파일명과 URL을 분리할 수 있다.
- 글 상세: `/posts/{slug}`
- 프로젝트 상세: `/projects/{slug}`
- 카테고리: `/categories/{category-slug}`
- 태그: `/tags/{tag-slug}`
- 한글 카테고리·태그명은 URL에는 영문 슬러그를, 화면에는 한글 표시명을 사용한다. (예: `학습 메모` → `/categories/learning-notes`)
- 슬러그는 한 번 배포되면 임의로 변경하지 않는 것을 원칙으로 한다.

### 6.4 읽기 시간 계산 기준

- 한국어 본문 기준 분당 약 500자(공백 포함)를 읽는 속도로 계산한다.
- 코드 블록은 본문 글자 수 계산에서 제외하고, 코드 블록 1개당 별도로 15초를 더한다.
- 계산 결과는 소수점 이하를 올림하여 "N분" 단위로 표시하며, 최소 표시값은 "1분"으로 한다.
- 계산은 빌드 시점에 본문 Markdown/MDX를 파싱하여 자동으로 생성하고, frontmatter에 별도 입력하지 않는다.

## 9. 기술 결정과 `AGENTS.md`의 범위

## 10. 검수 대상과 완료 조건

본 문서는 검수해야 할 결과와 완료 상태만 정의한다. 테스트 도구, 실행 명령, 세부 QA 시나리오, 검사 순서와 증거 수집 방식은 별도 `AGENTS.md`에서 정의한다.

### 10.1 검수 대상

- 게시글이 없는 상태에서도 홈과 탐색 페이지가 정상적으로 표시되어야 한다.
- 게시글을 추가하면 최신순 정렬, 카테고리와 태그 분류가 정확하게 반영되어야 한다.
- 프로젝트 목록에서 개발 블로그 상세 페이지로 이동할 수 있어야 한다.
- 값이 없는 저장소와 배포 링크는 화면에 나타나지 않아야 한다.
- 라이트 모드와 다크 모드가 작동하고 사용자의 선택이 유지되어야 한다.
- 모바일, 태블릿, 데스크톱에서 핵심 콘텐츠와 기능을 사용할 수 있어야 한다.
- 긴 제목, URL, 코드와 표가 콘텐츠 이용을 방해하지 않아야 한다.
- 키보드 탐색, 문서 구조, 대체 텍스트와 명암 대비 요구사항을 충족해야 한다.
- 로컬 실행과 프로덕션 빌드가 오류 없이 완료되어야 한다.
- 외부 배포나 서비스 연결 없이 로컬에서 전체 결과를 확인할 수 있어야 한다.

### 10.2 최종 완료 정의

다음 조건을 모두 만족하면 현재 프로젝트 범위를 완료한 것으로 본다.

1. 네 단계의 결과물이 모두 사용자 승인을 받는다.
2. 필수 페이지와 로컬 MVP 기능이 구현된다.
3. 게시글 없이도 홈이 정상적으로 완성되어 보인다.
4. 개발 블로그 프로젝트 상세 페이지가 실제 정보로 작성된다.
5. 별도 `AGENTS.md`에 정의된 QA가 완료되고 결과가 보고된다.
6. 프로덕션 빌드가 성공한다.
7. 외부에는 배포되지 않은 상태로 로컬 최종본이 전달된다.
