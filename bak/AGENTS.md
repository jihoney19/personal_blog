# 개인 개발 블로그 Agent 작업 지침

## 1. 문서 목적과 적용 범위

이 문서는 이 저장소에서 작업하는 모든 AI Agent의 구현, 검증, 보고 기준을 정의한다. 저장소 루트와 모든 하위 경로에 적용하며, 하위 폴더에 더 구체적인 `AGENTS.md`가 생기면 해당 폴더에서는 하위 문서가 우선한다.

이 문서는 사용자 승인 시점부터 구현과 QA의 기준으로 사용한다. 아직 승인되지 않은 제안은 구현된 요구사항으로 간주하지 않는다.

## 2. 기준 문서와 우선순위

작업 시작 전에 변경 범위와 관련된 아래 문서를 읽는다.

1. 사용자의 현재 지시
2. `개발블로그_기획문서_v1.0.md`
3. `개발블로그_디자인기획문서_v1.0.md`
4. 이 `AGENTS.md`

문서별 책임은 다음과 같다.

- 서비스·기능 기획 문서: 목표, 기능 범위, 콘텐츠 구조, 완료 조건
- 디자인 기획 문서: 시각 체계, 레이아웃, 반응형, 컴포넌트 상태, 접근성 표현
- `AGENTS.md`: 작업 절차, 코드와 파일 규칙, 기술 선택, 테스트와 결과 보고

요구사항이 충돌하면 임의로 섞지 않는다. 사용자의 최신 지시가 최우선이며, 두 기획 문서 사이에서는 디자인 문서의 `0.1 관련 문서 정합성 메모`에 명시된 최신 결정을 우선한다. 따라서 현재 MVP에서는 다음을 적용한다.

- 콘텐츠 유형은 글의 형식이고, 카테고리는 `AI Agent`, `프론트엔드`, `백엔드`, `Astro` 같은 기술 주제다.
- 외부 연락 수단은 실제 주소가 제공된 GitHub만 사용하며 이메일과 연락하기 기능은 제외한다.

그 밖의 충돌이 결과나 범위를 크게 바꾸면 구현을 멈추고 사용자에게 결정이 필요한 항목만 간결하게 보고한다.

## 3. 프로젝트 원칙

- Astro 기반 정적 사이트로 구현한다.
- 게시글과 프로젝트는 로컬 Markdown 또는 MDX와 Astro Content Collections로 관리한다.
- 기본 언어는 한국어다. URL 슬러그, 코드 식별자와 개발용 파일명은 영문을 사용한다.
- 초기 실제 사이트에는 게시글을 넣지 않는다. 샘플 글과 가짜 학습 기록을 만들지 않는다.
- 첫 프로젝트인 `개인 개발 블로그`에는 사용자가 확인한 실제 정보만 기록한다.
- 이름, 경력, 성과, 날짜, 수치, 연락처, 저장소 주소, 배포 주소와 이미지를 추측하지 않는다.
- 외부 API, 데이터베이스, CMS와 런타임 네트워크 요청 없이 로컬에서 완전히 빌드되어야 한다.
- 정적 HTML과 CSS를 기본으로 하고, 브라우저 JavaScript는 테마, 모바일 메뉴, 목차, 코드 복사처럼 상호작용에 필요한 경우에만 추가한다.
- 접근성, 반응형과 성능은 마무리 작업이 아니라 컴포넌트 구현의 완료 조건이다.

## 4. Agent의 자율 작업 범위

### 4.1 별도 승인 없이 진행할 수 있는 작업

승인된 기획과 현재 작업 목표 안에서는 다음을 자율적으로 수행한다.

- 폴더와 파일 생성, 코드 구현, 리팩터링과 오류 수정
- 콘텐츠 스키마, 타입, 재사용 컴포넌트와 순수 유틸리티 작성
- 로컬 개발용 의존성 추가와 설정 변경
- 단위 테스트, E2E 테스트, 접근성 검사와 테스트 fixture 작성
- 로컬 서버 실행, 프로덕션 빌드, 브라우저 검증과 화면 캡처
- 변경 범위에 필요한 기술 문서와 QA 결과 갱신

작업 중 사소한 구현 선택은 질문으로 넘기지 말고 이 문서와 기획 의도에 맞는 가장 단순한 방법을 선택한다.

### 4.2 사용자 승인이 필요한 작업

다음 작업은 실행 전에 사용자의 명시적 승인을 받는다.

- MVP 기능 범위의 확대 또는 필수 기능의 삭제
- 외부 배포, 도메인 연결, 저장소 공개 또는 원격 저장소 push
- 유료 제품, 유료 API, 유료 글꼴 또는 지속 비용이 생기는 서비스 사용
- 데이터베이스, CMS, 분석 도구, 댓글, 인증 또는 외부 백엔드 연결
- 개인정보, 연락처, 비밀정보 또는 보안 정책에 영향을 주는 변경
- 사용자의 실제 경력, 성과나 프로젝트 기여 범위를 확정하여 공개 콘텐츠로 반영하는 작업
- Astro 이외의 UI 프레임워크 도입이나 핵심 기술 스택 변경

## 5. 표준 작업 순서

1. 관련 기획 문서와 현재 저장소 상태를 확인한다.
2. 작업을 기능, 디자인, 콘텐츠, 설정 또는 오류 수정으로 분류하고 완료 조건을 정한다.
3. 디자인 시안 승인이 필요한 화면이면 승인된 시안이 있는지 확인한다. 없으면 실제 UI 구현 전에 시안 단계의 결과를 먼저 준비한다.
4. 기존 코드와 테스트를 보존하며 요청을 충족하는 가장 작은 일관된 변경을 구현한다.
5. 구현과 동시에 정상 상태, 빈 상태, 오류 상태와 경계 조건을 처리한다.
6. 변경 범위에 맞는 정적 검사와 자동 테스트를 실행한다.
7. 실제 브라우저에서 기능, 키보드, 테마와 반응형 레이아웃을 확인한다.
8. 실패 원인을 수정한 뒤 영향 범위에 맞는 회귀 테스트를 다시 실행한다.
9. 변경 내용, 검증 결과, 남은 제약과 사용자 확인 항목을 보고한다.

검증을 생략하고 완료라고 보고하지 않는다. 도구나 환경 문제로 검증하지 못한 항목은 실패와 구분해 정확한 이유를 남긴다.

## 6. 기술 기준

### 6.1 기본 도구

- 패키지 관리자는 `npm` 하나만 사용하고 `package-lock.json`을 커밋 대상으로 유지한다.
- 프로젝트 생성 시 선택한 Astro와 Node.js 버전은 `package.json`의 `engines`와 버전 파일에 기록한다.
- Astro는 정적 출력으로 명시하고 서버 adapter를 추가하지 않는다.
- TypeScript를 사용하고 Astro의 `strict` 설정을 확장한다.
- 코드 진단은 `astro check`, ESLint와 Prettier를 사용한다.
- 순수 로직의 단위 테스트는 Vitest, 실제 페이지의 E2E 테스트는 Playwright를 사용한다.
- 자동 접근성 검사는 Playwright와 axe를 사용하고, 자동화할 수 없는 항목은 키보드와 화면 확대를 직접 검수한다.

### 6.2 패키지 추가 기준

패키지를 추가하기 전에 다음 순서로 판단한다.

1. Astro, 브라우저 표준 API 또는 기존 코드로 간단하고 안전하게 해결할 수 있는가?
2. 정적 사이트의 번들 크기와 클라이언트 JavaScript를 불필요하게 늘리지 않는가?
3. 유지보수 상태, 라이선스, 보안 이슈와 Astro 호환성을 확인했는가?
4. 같은 역할의 패키지가 이미 설치되어 있지 않은가?
5. 프로덕션 의존성과 개발 의존성을 올바르게 구분했는가?

UI 프레임워크, 상태 관리 라이브러리, CSS 프레임워크와 범용 유틸리티 라이브러리는 편의만을 이유로 추가하지 않는다. 새 패키지를 추가하면 목적과 영향, 검증 결과를 작업 보고에 포함한다. 버전 범위를 바꾸거나 의존성을 갱신한 경우 전체 QA를 실행한다.

## 7. 권장 프로젝트 구조

Astro 프로젝트를 만들 때 다음 구조를 기본으로 사용한다. 기능상 이유가 있을 때만 구조를 확장하고, 같은 목적의 파일을 여러 위치에 흩어 놓지 않는다.

```text
/
├── AGENTS.md
├── astro.config.mjs
├── package.json
├── package-lock.json
├── tsconfig.json
├── eslint.config.js
├── playwright.config.ts
├── vitest.config.ts
├── public/
│   └── 정적 복사만 필요한 favicon·font 파일
├── src/
│   ├── assets/                 # Astro가 처리할 이미지와 로컬 자산
│   ├── components/
│   │   ├── content/            # 글 목록, 목차, 관련 글, 코드 블록
│   │   ├── layout/             # 헤더, 푸터, 페이지 프레임
│   │   ├── project/            # 프로젝트 전용 컴포넌트
│   │   └── ui/                 # 버튼, 배지, 빈 상태 등 공통 UI
│   ├── content/
│   │   ├── posts/              # 실제 게시글 Markdown·MDX
│   │   └── projects/           # 실제 프로젝트 Markdown·MDX
│   ├── content.config.ts       # 컬렉션 loader와 스키마
│   ├── data/                   # 검증된 사이트·프로필 설정
│   ├── layouts/                # 페이지와 Markdown 레이아웃
│   ├── lib/                    # 정렬, 분류, 읽기 시간 등 순수 로직
│   ├── pages/                  # 파일 기반 라우트
│   ├── scripts/                # 꼭 필요한 소규모 브라우저 스크립트
│   └── styles/
│       ├── tokens.css          # 색상, 간격, 글꼴과 레이아웃 토큰
│       └── global.css          # reset, 전역 본문과 접근성 기본값
└── tests/
    ├── e2e/
    ├── fixtures/               # 실제 사이트에 노출되지 않는 검증 데이터
    └── unit/
```

- 현재 루트의 승인 문서는 임의로 이동하거나 이름을 바꾸지 않는다.
- `public/`에는 빌드 변환이 필요 없는 파일만 둔다. 최적화할 이미지는 `src/assets/`에 둔다.
- 테스트 fixture는 `tests/fixtures/`에 두고 일반 빌드가 읽지 않게 한다.
- 생성물인 `dist/`, `.astro/`, `coverage/`, `playwright-report/`, `test-results/`와 임시 QA 캡처는 버전 관리에서 제외한다.

## 8. 콘텐츠와 데이터 규칙

### 8.1 Content Collections

- `src/content.config.ts`에서 `posts`와 `projects` 컬렉션을 명시적으로 정의한다.
- 로컬 Markdown·MDX는 Astro의 로컬 glob loader로 불러오고 모든 frontmatter를 스키마로 검증한다.
- 날짜, URL, 열거형과 배열은 느슨한 문자열로 넘기지 말고 스키마에서 제약한다.
- 스키마 오류는 빌드 실패로 처리한다. 임의 기본값으로 잘못된 실제 콘텐츠를 숨기지 않는다.
- 초안 게시글은 목록, 분류 페이지, 관련 글과 정적 상세 페이지 생성 대상에서 모두 제외한다.

### 8.2 게시글

- 필수 필드: `title`, `description`, `publishedAt`, `type`, `category`, `tags`, `draft`
- 선택 필드: `updatedAt`, `status`
- 날짜는 `YYYY-MM-DD` 형식을 기본으로 하고, 정렬 시 시간대에 따라 날짜가 바뀌지 않게 처리한다.
- 최신순 정렬의 동률은 영문 ID 오름차순으로 고정해 빌드마다 결과가 바뀌지 않게 한다.
- 슬러그와 파일명은 영문 소문자, 숫자와 하이픈만 사용한다.
- 태그는 빈 배열을 허용하되 중복과 앞뒤 공백은 허용하지 않는다.

### 8.3 프로젝트

- 필수 필드: `title`, `summary`, `status`, `startedAt`, `role`, `stack`
- 선택 필드: `endedAt`, `repoUrl`, `demoUrl`, `relatedPosts`
- 선택 링크 값이 없으면 행, 버튼과 안내 문구를 모두 렌더링하지 않는다.
- 종료일은 시작일보다 빠를 수 없다.
- 관련 글 참조가 있으면 실제로 존재하며 공개 가능한 게시글인지 검증한다.

### 8.4 실제 콘텐츠와 fixture 분리

- 빈 상태는 실제 `posts` 폴더가 비어 있는 일반 빌드에서 검증한다.
- 게시글이 있는 상태, 긴 제목, 코드, 표, 관련 글과 이전·다음 글은 `tests/fixtures/`의 명백한 테스트 데이터로 검증한다.
- E2E는 `empty`와 `fixture` 두 프로필을 분리해 서로 다른 포트와 빌드 출력 경로에서 실행한다.
- fixture loader는 테스트 프로필에서만 활성화하고 일반 `npm run build`에서는 접근할 수 없게 한다.
- fixture를 실제 콘텐츠 폴더로 복사하거나 최종 빌드 결과에 포함하지 않는다.
- fixture에 실제 개인정보나 비밀값을 사용하지 않는다.

## 9. 코드 작성 규칙

### 9.1 TypeScript와 일반 형식

- 들여쓰기는 공백 2칸으로 하고 최종 형식은 Prettier 설정을 따른다.
- `any`, 무근거한 타입 단언, `@ts-ignore`와 불필요한 non-null assertion을 사용하지 않는다.
- 외부 입력과 frontmatter는 경계에서 검증하고 내부 로직은 구체적인 타입을 사용한다.
- 동일 로직을 페이지마다 복사하지 말고 `src/lib/`의 순수 함수로 분리한다.
- 사용하지 않는 코드, 주석 처리한 코드와 디버그 로그를 남기지 않는다.
- 주석은 코드가 무엇을 하는지 반복하지 말고, 선택 이유나 피할 수 없는 제약을 설명할 때만 작성한다.

### 9.2 명명

- Astro 컴포넌트와 타입: `PascalCase`
- 함수, 변수와 props: `camelCase`
- 상수: 값이 진정한 전역 불변 상수일 때만 `UPPER_SNAKE_CASE`
- 라우트, 콘텐츠 파일, CSS 클래스와 일반 자산: `kebab-case`
- 불리언: `is`, `has`, `can`, `should`처럼 상태가 드러나는 접두어 사용
- 이벤트 함수: `handleCopy`, `handleMenuToggle`처럼 동작이 드러나는 이름 사용

### 9.3 Astro 컴포넌트

- 컴포넌트는 한 가지 시각적 또는 행동 책임을 갖게 한다.
- props는 `interface Props`로 선언하고 선택 여부와 기본값을 명확히 한다.
- 기본은 서버 렌더링되는 `.astro` 컴포넌트다. 프레임워크 island는 승인된 필요가 없으면 추가하지 않는다.
- 의미론적 HTML을 우선하며 클릭 가능한 `div`, 중첩 링크와 잘못된 heading 순서를 만들지 않는다.
- 카드 전체가 링크인 경우 하나의 기본 링크 구조로 구현하고 내부에 다른 링크를 중첩하지 않는다.
- 조건부 값이 없을 때 빈 래퍼, 빈 구분점, 비활성 버튼과 `준비 중` 가짜 링크를 남기지 않는다.

### 9.4 CSS와 디자인 토큰

- 디자인 문서의 값은 `tokens.css`의 의미 기반 CSS custom property로 중앙화한다.
- 컴포넌트에서 승인된 색상, 간격과 글꼴 값을 임의의 숫자로 반복하지 않는다.
- mobile-first를 기본으로 하되 디자인 문서의 정확한 전환 지점을 지킨다.
- `:focus-visible`, `prefers-reduced-motion`, 200% 확대와 가로 overflow를 모든 관련 컴포넌트에서 고려한다.
- 전역 선택자는 reset, 토큰, 기본 타이포그래피와 Markdown 본문처럼 실제 전역 규칙에만 사용한다.
- 그림자, 그라데이션, 유리 효과, 큰 모서리 반경과 장식용 애니메이션을 추가하지 않는다.

### 9.5 브라우저 JavaScript

- 기능을 작은 모듈로 분리하고 전역 상태와 전역 이벤트 누적을 피한다.
- 이벤트 리스너는 중복 등록되지 않아야 하며 필요한 경우 정리한다.
- local storage와 clipboard 사용은 실패 가능성을 처리한다.
- 테마는 첫 화면이 그려지기 전에 저장된 선택 또는 운영체제 설정을 적용해 깜빡임을 방지한다.
- 메뉴, 목차와 복사 상태는 화면 표시뿐 아니라 적절한 ARIA 상태 또는 live region으로 전달한다.

## 10. 필수 npm 명령

프로젝트가 생성되면 `package.json`에 아래 역할의 명령을 제공한다. 명령 이름을 바꿔야 할 이유가 있으면 이 문서도 함께 갱신한다.

| 명령 | 목적 |
|---|---|
| `npm ci` | lockfile과 정확히 일치하는 의존성 설치 |
| `npm run dev` | 로컬 개발 서버 실행 |
| `npm run build` | 정적 프로덕션 빌드 생성 |
| `npm run preview` | 빌드 결과 로컬 미리보기 |
| `npm run check` | Astro와 TypeScript 진단, warning 이상 실패 처리 |
| `npm run lint` | ESLint 정적 검사 |
| `npm run format:check` | Prettier 형식 검사 |
| `npm run test:unit` | Vitest 단위 테스트 1회 실행 |
| `npm run test:e2e` | empty·fixture 프로덕션 preview 대상 Playwright 테스트 |
| `npm run test:a11y` | 자동 접근성 시나리오 실행 |
| `npm run qa` | check, lint, format, unit, E2E, 최종 일반 build 전체 검증 |

- `test:e2e`를 단독 실행해도 empty와 fixture용 정적 빌드와 preview가 각각 준비되도록 Playwright `webServer` 또는 동등한 오케스트레이션을 둔다.
- fixture 빌드는 별도 출력 경로를 사용하고, `qa`의 마지막에는 fixture가 없는 일반 `npm run build`를 다시 실행한다.
- 테스트는 개발 서버보다 프로덕션 preview를 대상으로 실행한다.
- 패키지를 일회성으로 내려받는 미고정 `npx` 명령을 정식 검증 절차로 사용하지 않는다.
- formatter의 자동 수정은 구현 중 사용할 수 있지만, 최종 검증은 변경을 만들지 않는 검사 명령으로 수행한다.

## 11. 자동 테스트 기준

### 11.1 단위 테스트

최소한 다음 순수 로직과 데이터 경계를 테스트한다.

- 게시글 공개 여부와 최신순 정렬, 동률 정렬
- 카테고리·태그 집계와 정렬
- 이전 글·다음 글 계산
- 관련 글에서 현재 글 제외, 같은 태그 우선, 카테고리 보충, 최대 3개 제한
- 예상 읽기 시간의 경계값
- 선택 필드와 링크의 조건부 표시 판단
- 잘못된 날짜, URL, 상태와 frontmatter에 대한 스키마 실패

버그를 수정할 때는 가능하면 먼저 실패를 재현하는 테스트를 추가하고, 수정 후 그 테스트가 통과하는지 확인한다.

### 11.2 E2E 테스트

최소한 다음 사용자 흐름을 검증한다.

- 게시글이 없는 홈에서 히어로와 빈 상태가 보이고 카테고리 필터는 숨겨진다.
- fixture 모드에서 게시글이 최신순으로 나타나고 카테고리 필터가 동작한다.
- 카테고리·태그 인덱스의 개수와 상세 목록이 정확하다.
- 프로젝트 목록에서 상세로 이동하며 없는 외부 링크는 렌더링되지 않는다.
- 테마가 운영체제 설정을 따르고, 사용자 선택 후 새로고침해도 유지된다.
- 모바일 메뉴가 열리고 닫히며 `Esc`, 포커스 복귀와 펼침 상태가 올바르다.
- 목차가 화면 폭에 따라 sticky 또는 접이식으로 바뀌고 제목 링크가 이동한다.
- 코드 복사 성공·실패 피드백이 화면과 보조기기에 전달된다.
- 이전·다음 글과 관련 글은 존재하는 항목만 표시한다.
- 404 페이지에서 홈으로 돌아갈 수 있다.
- 모든 내부 링크가 유효하고 브라우저 console error가 없다.

E2E selector는 스타일 클래스나 DOM 순서보다 role, label, heading과 안정적인 `data-testid`를 사용한다. `data-testid`는 사용자 관점 selector로 구분할 수 없는 경우에만 추가한다.

## 12. 수동 QA 체크리스트

### 12.1 브라우저와 반응형

- 개발 중 영향 범위 검증은 Chromium에서 수행한다.
- 최종 QA는 Chromium, Firefox와 WebKit에서 수행한다.
- 기본 화면 검수 폭은 `360px`, `768px`, `1280px`다.
- 레이아웃 전환을 수정한 경우 `640/641`, `768/769`, `850/851`, `1023/1024`, `1279/1280px` 경계도 확인한다.
- 각 주요 페이지를 라이트·다크 모드에서 확인한다.
- 긴 제목, 긴 URL, 코드, 표와 이미지 때문에 페이지 전체에 가로 스크롤이 생기지 않는지 확인한다.

### 12.2 접근성

- WCAG 2.2 AA를 목표로 axe의 serious 또는 critical 위반을 0건으로 유지한다.
- 일반 텍스트 4.5:1, 큰 텍스트와 UI 경계 3:1 이상의 대비를 확인한다.
- 키보드만으로 skip link, 메뉴, 테마, 필터, 목차, 복사와 모든 링크를 사용할 수 있어야 한다.
- 포커스가 가려지거나 사라지지 않고 시각적 순서와 이동 순서가 일치해야 한다.
- 페이지마다 H1은 하나이며 heading 수준을 건너뛰지 않는다.
- 의미 있는 이미지는 구체적인 대체 텍스트를 갖고 장식 이미지는 보조기기에서 제외한다.
- 200% 확대, reduced motion과 색상 제거 상황에서도 정보와 기능이 유지되어야 한다.
- 색상만으로 현재 위치, 상태, 성공과 실패를 전달하지 않는다.

### 12.3 성능과 정적 결과

- 프로덕션 preview의 대표 페이지를 Lighthouse mobile 기준으로 검사한다.
- 목표 점수는 Performance 90 이상, Accessibility 100, Best Practices 95 이상, SEO 90 이상이다.
- 성능 목표를 넘지 못하면 원인을 기록하고 이미지, 글꼴, JavaScript와 렌더링 차단 자원을 먼저 점검한다.
- 빌드 결과에 서버 전용 코드, 테스트 fixture, source map, 비밀값과 불필요한 대형 자산이 포함되지 않았는지 확인한다.
- JavaScript가 비활성화되어도 문서 콘텐츠와 기본 탐색 링크는 읽고 이동할 수 있어야 한다. 상호작용 기능은 이해 가능한 기본 상태를 유지한다.

## 13. 변경별 회귀 테스트

| 변경 유형 | 최소 회귀 범위 |
|---|---|
| 문서만 변경 | 링크, 용어와 상충 여부 확인 |
| 순수 유틸리티 | 관련 단위 테스트, check, lint, format |
| 콘텐츠·스키마 | 스키마 실패 테스트, 정렬·분류 단위 테스트, build |
| 개별 컴포넌트 | 관련 단위/E2E, 키보드, 라이트·다크, 360·1280px |
| 공통 레이아웃·토큰 | 전체 주요 페이지, 모든 기본 폭, 두 테마, 접근성 검사 |
| 탐색·테마·목차·복사 | 관련 E2E, 키보드, 상태 전달, 3개 브라우저 |
| Astro·패키지·빌드 설정 | `npm run qa` 전체, 빌드 결과 점검 |
| 오류 수정 | 재현 테스트, 직접 영향 기능, 공유 의존 경로의 회귀 테스트 |

테스트를 통과시키기 위해 assertion, 접근성 규칙, 타입 검사나 lint 규칙을 근거 없이 약화하지 않는다. flaky 테스트는 재실행만으로 넘기지 말고 원인을 찾아 안정화한다.

## 14. 보안, 개인정보와 금지 작업

- `.env`, 토큰, 비밀번호, 개인 이메일과 비공개 URL을 소스나 캡처에 기록하지 않는다.
- 브라우저에 노출 가능한 값만 `PUBLIC_` 환경 변수로 사용한다.
- `.env.example`이 필요하면 값이 아닌 변수명과 설명만 제공한다.
- 라이선스가 확인되지 않은 폰트, 이미지, 아이콘과 코드를 저장소에 복사하지 않는다.
- Markdown의 raw HTML과 임의 script 실행은 기본적으로 허용하지 않는다.
- 외부 링크는 실제 값이 있을 때만 렌더링하고 새 창 링크에는 안전한 `rel` 값을 제공한다.
- 사용자 승인 없이 배포 명령, 원격 push, 도메인·분석·댓글·CMS 연결을 실행하지 않는다.
- 테스트를 위해 실제 사용자 정보, 가짜 경력, 가짜 프로젝트 성과 또는 실제처럼 보이는 게시글을 만들지 않는다.
- 범위를 벗어난 기능을 `나중에 필요할 수 있다`는 이유로 미리 구현하지 않는다.
- 실패를 숨기기 위해 오류를 삼키거나 빈 `catch`, 무조건 성공하는 fallback을 추가하지 않는다.

## 15. 작업 결과 보고 형식

작업을 마치면 사용자가 코드를 직접 읽지 않아도 결과를 판단할 수 있도록 다음 순서로 보고한다.

1. **결과 요약**: 무엇이 완료되었는지 1~3문장
2. **주요 변경**: 사용자에게 보이는 동작과 중요한 기술 변경
3. **검증 결과**: 실행한 명령 또는 수동 시나리오와 성공·실패·미실행 상태
4. **화면 QA**: 확인한 브라우저, 화면 폭, 테마와 필요한 캡처 경로
5. **남은 항목**: 알려진 제약, 미검증 항목과 그 이유
6. **사용자 확인**: 실제 개인정보, 콘텐츠 사실 또는 큰 단계 승인처럼 사용자만 결정할 수 있는 항목

검증 결과에는 `모두 통과` 같은 요약만 쓰지 말고 실행한 검사와 대상 범위를 함께 적는다. 실패가 있으면 실패한 항목, 관찰된 영향과 다음 조치를 분리해 보고한다.

## 16. 완료 정의

개별 작업은 다음 조건을 모두 만족해야 완료다.

- 요청과 승인된 기획 범위를 충족한다.
- 정상, 빈 상태, 오류와 경계 조건을 처리한다.
- 코드, 콘텐츠와 UI가 이 문서의 규칙을 따른다.
- 변경 범위에 필요한 자동 테스트와 수동 QA를 통과한다.
- 프로덕션 빌드가 성공하고 새 warning과 console error가 없다.
- 접근성, 반응형과 두 테마에서 기능 손실이 없다.
- 실제 사이트에 fixture, 가짜 정보, 비밀값과 불필요한 생성물이 포함되지 않는다.
- 변경 내용과 검증 증거가 사용자 관점으로 보고된다.

프로젝트의 로컬 MVP와 최종본 완료 조건은 이 문서와 두 기획 문서의 완료 조건을 모두 만족해야 한다.
