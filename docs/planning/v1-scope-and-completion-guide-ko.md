# DEV.LOG v1 종료 범위 및 완료 가이드

> 기준일: 2026-07-18
> 문서 목적: Personal Development Blog의 v1 개발을 어디까지 진행하고 언제 종료할지 판단하는 기준을 정의한다.

## 1. v1의 목표

v1은 기능을 계속 추가하는 단계가 아니라, 실제 글과 프로젝트 정보를 안전하게 등록하고 독자가 읽을 수 있는 정적 개발 블로그를 완성하는 단계다.

v1 종료 시점에는 다음이 가능해야 한다.

- Markdown 글을 추가하면 목록·상세·카테고리·태그 페이지가 자동으로 생성된다.
- 프로젝트와 About 페이지가 실제로 확인된 정보만 표시한다.
- 모바일과 데스크톱에서 글을 읽고 탐색할 수 있다.
- 라이트·다크 테마, TOC, 코드 복사, 이전·다음 글, 관련 글이 정상 동작한다.
- 초안·잘못된 링크·누락된 필드가 공개 결과에 섞이지 않는다.
- 정적 빌드와 브라우저 QA가 완료된다.

v1 이후에도 글을 작성하고 운영하는 것은 계속되지만, 위 조건을 충족한 시점에 “블로그 개발 프로젝트 v1”은 종료한다.

## 2. v1에 포함하는 범위

### 2.1 콘텐츠와 페이지

- Home: 블로그 소개, 최신 글, 카테고리 필터, 글이 없을 때의 빈 상태
- Logs: 공개 글 전체 목록과 카테고리 필터
- Post Detail: 제목, 설명, 날짜, 읽기 시간, 본문, 태그, TOC, 코드 블록, 이전·다음 글, 관련 글
- Categories: 카테고리 목록·개수·상세 목록
- Tags: 태그 목록·개수·상세 목록
- Projects: 프로젝트 목록과 프로젝트 상세
- About: 실제 확인된 이름·소개·기술 활용·AI 에이전트 협업·GitHub 링크
- 404: Home으로 돌아가는 안내

### 2.2 상호작용

- 운영체제 설정을 따르는 초기 테마
- 사용자가 선택한 테마의 저장과 새로고침 후 복원
- 모바일 메뉴 열기·닫기·`Esc` 처리·포커스 복귀
- 모바일 TOC 접기·펼치기와 데스크톱 sticky TOC
- 현재 읽는 섹션의 TOC 상태 표시
- 코드 복사 성공·실패 상태의 시각·보조기술 안내
- 카테고리 필터의 선택 상태와 결과 없음 상태
- 댓글 기능 플래그가 켜진 환경에서의 로딩·빈 목록·등록 성공·실패 상태

### 2.3 공개 품질과 기본 SEO

- 페이지별 고유 title과 description
- canonical URL
- Open Graph 기본 메타데이터
- `robots.txt`와 sitemap
- 공개 글만 검색 엔진과 RSS 대상에 포함
- 내부 링크와 post slug의 유효성 검증

대표 이미지가 실제로 준비되지 않은 경우에는 가짜 이미지를 만들지 않고 기본 공유 이미지 또는 이미지 없는 상태로 처리한다.

### 2.4 테스트와 QA

- `npm run check`
- `npm run lint`
- `npm run format:check`
- `npm run test:unit`
- `npm run test:e2e`
- `npm run test:a11y`
- `npm run build`
- 최종 정적 빌드에 초안·fixture·비밀값·서버 전용 코드가 포함되지 않는지 확인
- Chromium, Firefox, WebKit에서 주요 흐름 확인
- 360px, 768px, 1280px에서 라이트·다크 모드 확인
- 키보드 탐색, 200% 확대, `prefers-reduced-motion`, 가로 넘침 확인
- 브라우저 콘솔 오류와 끊어진 내부 링크 확인

## 3. v1에 포함하지 않는 범위

다음 항목은 v1 종료를 늦추지 않고 후속 버전으로 관리한다.

- 전체 텍스트 검색
- 연·월별 아카이브
- 시리즈·학습 경로
- 자동 OG 이미지 생성
- 좋아요·반응 기능
- 이메일 구독과 뉴스레터
- 방문자 분석과 쿠키 동의 흐름
- 로그인·계정·관리자 페이지
- 댓글의 인증 계정, 답글, 수정·삭제
- 다국어 사이트
- PWA·오프라인 모드
- 도메인 구매와 외부 배포 자동화의 추가 확장

댓글의 로컬 코드와 개발 Supabase 검증은 v1 범위에 포함할 수 있다. 다만 실제 운영 프로젝트 연결과 공개 운영은 별도의 배포 승인과 운영 점검을 통과해야 한다.

## 4. 완료 판정 기준

다음 조건을 모두 만족하면 v1을 완료로 판정한다.

### 기능

- 공개 글과 초안의 노출 경계가 정확하다.
- 날짜 정렬, 동률 시 ID 정렬, 카테고리·태그 개수가 정확하다.
- 이전·다음 글과 관련 글이 존재하는 경우에만 표시된다.
- 선택적 URL과 업데이트 날짜가 없을 때 빈 행·빈 버튼이 생기지 않는다.
- 글이 하나도 없는 상태에서도 Home이 오류 화면처럼 보이지 않는다.
- 존재하지 않는 경로는 404에서 복구할 수 있다.

### 접근성

- 페이지마다 H1이 하나이고 heading 순서를 건너뛰지 않는다.
- 모든 주요 기능을 키보드만으로 사용할 수 있다.
- 포커스가 숨겨지지 않고 `:focus-visible` 상태가 보인다.
- 상태·오류·성공 메시지를 색상만으로 전달하지 않는다.
- 의미 있는 이미지에는 대체 텍스트가 있고, 장식 이미지는 보조기술에서 제외된다.
- axe 검사에서 serious 또는 critical 위반이 없다.

### 정적 출력과 운영

- 일반 `npm run build`가 fixture 없이 성공한다.
- 빌드 결과에 source map, 테스트 산출물, 비밀값이 들어 있지 않다.
- 실제 개인 정보·프로젝트 사실·외부 URL이 사용자 검수를 통과했다.
- 새 Markdown 파일을 추가하는 절차가 README 또는 운영 문서에 설명되어 있다.
- 발행 전 `BLOG_POSTING.md`의 승인 절차를 따를 수 있다.

## 5. 권장 마감 순서

1. 기획 문서와 구현의 필드명·표시 규칙을 맞춘다.
2. About, 프로젝트 역할·기간·링크 등 실제 사실을 사용자에게 검수받는다.
3. canonical, Open Graph 기본값, sitemap, robots를 마무리한다.
4. 전체 정적 빌드와 테스트를 실행한다.
5. 세 브라우저·세 기준 폭·두 테마에서 수동 QA를 수행한다.
6. 발견된 문제를 수정하고 영향 범위에 맞는 회귀 테스트를 다시 실행한다.
7. v1 완료 문서와 남은 항목을 기록한다.
8. 이후 기능은 v1.1 작업 목록으로 분리한다.

## 6. v1.1 후보

v1이 안정된 뒤 다음 순서로 검토한다.

1. 정적 전체 검색
2. 연·월별 아카이브
3. 시리즈·학습 경로
4. 자동 OG 이미지
5. 댓글 운영 데이터와 스팸 대응 개선

이 목록의 우선순위는 실제 글 수, 독자의 탐색 문제, 운영 부담을 확인한 뒤 조정한다.

## 7. 사용자 승인 필요 항목

다음은 코드만으로 확정할 수 없으며 사용자의 최종 확인이 필요하다.

- 공개할 이름과 소개 문구
- 프로필 이미지와 이미지 사용 권한
- GitHub URL
- 프로젝트의 실제 역할·기여 범위·개발 기간
- 프로젝트 저장소 및 배포 URL
- 댓글 기능의 실제 운영 연결 여부
- v1 공개 배포 여부와 배포 대상 URL

## 8. 종료 선언 문구

다음 문장을 만족하면 v1 종료를 선언한다.

> 실제 정보가 검수된 DEV.LOG가 정적 빌드로 생성되고, 주요 브라우저와 기준 화면 폭에서 접근 가능하며, 새 Markdown 글을 안전하게 발행할 수 있다.

v1 종료는 기능 개발의 영구적인 종료를 의미하지 않는다. 이후의 검색·아카이브·시리즈·분석 등의 작업은 운영 중 확인된 필요를 기준으로 v1.1 이상의 별도 범위로 계획한다.
