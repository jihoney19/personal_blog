[정적 개발 블로그의 기술 스택을 선택한 이유](/posts/technology-stack-for-a-static-development-blog/)에서는 Markdown 콘텐츠를 정적 HTML로 만드는 구조를 정리했습니다. 이번 단계에서는 그 구조를 유지하면서 독자가 남기는 댓글만 별도의 런타임 데이터로 분리했습니다.

## 정적 콘텐츠와 런타임 데이터 분리

게시글과 프로젝트는 계속 Git과 Markdown에서 관리합니다. 댓글은 독자가 생성하고 상태가 계속 바뀌므로 Supabase의 PostgreSQL 데이터베이스에 저장합니다. 이 경계를 명확히 두면 댓글 기능을 추가해도 블로그 본문은 외부 서비스 없이 빌드하고 읽을 수 있습니다.

댓글 기능은 다음 흐름으로 구성했습니다.

```text
브라우저 -> Edge Function -> comments 테이블
                         -> comment_rate_limits 테이블
브라우저 -> RLS가 적용된 approved 댓글 조회
```

브라우저에는 프로젝트 URL만 전달하고, Supabase 인증 키와 해시 비밀값은 Edge Function 환경에만 둡니다. 브라우저는 출처 검증과 요청 제한이 적용된 댓글 엔드포인트만 호출합니다.

## 승인형 댓글 스키마와 RLS

`comments` 테이블에는 게시글 식별자, 작성자명, 본문, 상태, 생성 시각과 승인 시각을 저장합니다. 상태는 `pending`, `approved`, `rejected`로 제한했습니다. 승인 상태로 바뀌면 트리거가 승인 시각을 자동으로 채웁니다.

공개 읽기 정책은 `approved` 상태만 허용합니다. 익명 사용자가 직접 `pending` 댓글을 조회하거나 테이블에 쓰는 경로는 열지 않았습니다. 댓글 작성은 검토된 Edge Function을 통해서만 처리합니다.

인덱스는 게시글별 승인 댓글을 시간순으로 조회하는 경로와 운영자가 상태별 댓글을 확인하는 경로에 맞췄습니다.

## 원자적 요청 제한

짧은 시간에 반복되는 댓글 요청을 제한하기 위해 `comment_rate_limits` 테이블과 PostgreSQL 함수를 추가했습니다. Edge Function은 원본 IP를 저장하지 않고 서버 비밀값으로 만든 해시만 함수에 전달합니다.

요청 제한 함수는 한 트랜잭션 안에서 현재 구간의 요청 수를 읽고 증가시킵니다. 여러 요청이 동시에 들어와도 각각 읽고 따로 저장하는 방식보다 제한을 일관되게 적용할 수 있습니다.

초기 함수에서는 `current_time`이라는 변수명이 PostgreSQL의 시간 관련 표현과 충돌해 RPC 호출이 실패했습니다. 변수명을 `request_time`으로 바꾸는 후속 마이그레이션을 추가하고, 저장소의 마이그레이션 기록과 개발 데이터베이스 상태를 일치시켰습니다.

## Edge Function을 쓰기 경계로 사용하기

댓글 작성 함수는 다음 검사를 담당합니다.

- 허용된 출처인지 확인
- 게시글 식별자, 작성자명, 본문 길이 검증
- 요청 제한 RPC 실행
- 댓글을 `pending` 상태로 저장
- 브라우저에 내부 오류나 비밀값을 노출하지 않는 응답 반환

승인되지 않은 댓글은 즉시 공개되지 않습니다. 정상 제출 뒤 화면에는 검토 대기 상태를 안내하고, 공개 목록은 계속 승인된 댓글만 보여줍니다.

## 댓글 UI와 실패 상태

게시글 상세 페이지에 댓글 목록과 작성 폼을 연결했습니다. 기능이 비활성화된 빌드, 댓글이 없는 상태, 목록 요청 실패, 제출 성공과 실패, JavaScript가 꺼진 상태를 각각 구분해 안내합니다.

로컬 개발에서는 Git에 포함되지 않는 환경 파일로 개발 프로젝트를 연결합니다. 배포 환경에 필요한 변수가 없으면 댓글 기능은 자동으로 비활성화되므로 개발 데이터베이스가 의도치 않게 프로덕션 사이트에 연결되지 않습니다.

## 검증

개발 프로젝트에 마이그레이션과 Edge Function을 적용한 뒤 다음 항목을 확인했습니다.

- 두 댓글 관련 테이블, 컬럼, 제약 조건 존재
- `comments` 테이블의 RLS 활성화
- 익명 사용자는 승인 댓글만 조회 가능
- 대기 댓글은 익명 조회 결과에서 제외
- 승인 시각 트리거와 요청 제한 RPC 존재
- 게시글마다 승인된 검증 댓글이 실제 브라우저에 표시
- 댓글 제출 결과가 `pending`으로 저장

저장소에서는 Astro 진단, ESLint, Prettier, Vitest와 프로덕션 빌드를 실행했습니다. Playwright로 Chromium, Firefox, WebKit에서 승인 댓글 조회, 빈 상태, 모바일 폼, 오류 후 재시도, JavaScript 비활성 상태를 확인했고, 댓글 영역의 심각 또는 치명적 접근성 위반이 없음을 검사했습니다.

## 마무리

이번 작업의 핵심은 정적 블로그 전체를 동적 애플리케이션으로 바꾸지 않고, 독자가 생성하는 데이터만 명확한 보안 경계 뒤에 배치한 것입니다. 스키마와 정책은 저장소의 마이그레이션으로 관리하고, 익명 읽기는 RLS로 제한하며, 쓰기와 요청 제한은 Edge Function과 원자적 RPC가 담당하도록 역할을 나눴습니다.
